# E:\power_bank 项目整体说明文档

## 1. 项目定位

`E:\power_bank` 是一个“共享充电宝管理系统”完整工程目录，不只是单独的前端仓库，而是由以下三部分组成：

1. `power/`
   Vue 2 + Element UI 前端项目，负责登录、注册、用户端页面、管理员后台页面。
2. `power_bank/`
   Flask 后端项目，负责认证、设备管理、租借归还、订单查询、充值、反馈处理、统计分析等业务接口。
3. `FISCO-BCOS/python-sdk/`
   区块链 SDK 依赖目录，供后端 `sysadmin` 模块调用，实现数据库与链上合约的数据同步。

从代码关系上看，这不是简单的“前后端分离但彼此独立”的项目，而是：

- 前端请求 Flask 后端接口
- 后端维护 MySQL 数据
- 后端还会在后台线程中和 FISCO-BCOS / WeBASE 进行链上同步

也就是说，整个系统是一个“前端页面 + 后端 API + MySQL + 区块链同步层”的组合架构。

---

## 2. 总体架构

整体调用关系如下：

```text
用户浏览器
  -> Vue 前端（E:\power_bank\power）
  -> HTTP / JSON
Flask 后端（E:\power_bank\power_bank）
  -> MySQL 数据库
  -> sysadmin 同步线程
  -> FISCO-BCOS / WeBASE
```

### 2.1 前端职责

前端负责：

- 登录、注册
- 用户查看附近设备并发起租借 / 归还
- 用户查看订单记录、结算、提交反馈、查看通知
- 用户编辑资料、充值、修改密码、注销账号
- 管理员管理设备、计费规则、纠纷工单、可视化大屏

### 2.2 后端职责

后端负责：

- 提供 REST 风格业务接口
- 连接 MySQL 执行 SQL
- 初始化数据库表结构
- 按模块注册蓝图
- 启动后台同步线程
- 与 WeBASE / FISCO-BCOS 交互

### 2.3 区块链层职责

区块链层不是给前端直接调用的，而是由后端 `sysadmin` 模块负责：

- 获取 WeBASE 中的系统管理员地址
- 同步合约 ABI、地址、字节码
- 将数据库中的管理员、用户、设备、计费规则等数据同步到链上
- 将链上部分数据反向同步回本地数据库

## 4. 前端项目说明：`E:\power_bank\power`

### 4.1 技术栈

前端 `package.json` 显示的核心技术如下：

- Vue 2.6
- Vue Router 3
- Vuex 3
- Element UI 2
- Axios
- ECharts 6
- Vue CLI 5

这是一个典型的 Vue 2 单页应用。

### 4.2 前端入口与基础配置

关键文件：

- `src/main.js`
  注册路由、Vuex、Element UI、Axios 插件，并挂载 `App.vue`
- `src/App.vue`
  以 `<router-view>` 为主，说明整个应用主要由路由驱动
- `src/router/index.js`
  定义登录注册、用户端、管理端路由
- `src/utils/request.js`
  统一 Axios 实例，开发环境默认请求 `<BACKEND_BASE_URL>`

目前前端认证上有两个特点：

- 请求拦截器会尝试从 `localStorage` 中读取 `token`
- 但当前登录接口并没有完整返回 token 闭环

这说明代码里对认证流程做了预留，但当前系统主要还是依赖 `localStorage` 中保存的用户信息和角色信息。

## 5. 后端项目说明：`E:\power_bank\power_bank`

### 5.1 技术栈

从 `requirements.txt` 和源码看，后端核心栈包括：

- Flask
- flask-cors
- PyMySQL
- DBUtils
- sqlparse
- selenium

后端不是 ORM 模式，而是：

- Flask + Blueprint
- 手写 SQL
- 自定义连接池封装

### 5.2 启动入口

启动文件是：

- `run.py`

核心逻辑位于 `app/__init__.py` 的 `create_app()` 中。

### 5.3 应用工厂 `app/__init__.py`

`create_app()` 做了这些关键事情：

1. 加载 Flask 环境配置
2. 加载 `instance/config.ini`
3. 初始化 MySQL 连接池
4. 注册认证、用户、管理员蓝图
5. 通过 `get_webase_address()` 读取系统管理员链上地址
6. 自动执行 `instance/power_bank.sql`
7. 启动后台同步线程

所以这个后端不是单纯的 API 服务，它启动时就会主动做数据库初始化和链上同步准备。

### 5.5 蓝图划分

#### 认证蓝图 `app/auth/routes.py`

负责：

- 用户注册 `POST /register`
- 登录 `POST /login`

特点：

- 普通用户注册时会写本地 `users` 表
- 同时调用链上用户服务创建对应链上用户
- `sysadmin` 登录会跳到 WeBASE 页面

#### 用户蓝图 `app/user/rent.py`

负责：

- 获取设备列表 `GET /api/powerbanks`
- 租借设备 `POST /api/rent`
- 归还设备 `POST /api/return`

特点：

- 查询支持分页、位置、状态、容量、功能筛选
- 租借时会校验押金余额
- 归还时会根据 `billing_rules` 计算费用

#### 用户蓝图 `app/user/query.py`

负责：

- 查询历史订单 `GET /api/history`
- 结算订单 `POST /api/history`
- 提交反馈 `POST /api/feedback`
- 获取通知 `GET /api/notifications`

特点：

- 前端所谓“结算”实际上也是走 `/api/history` 的 POST 分支
- 管理员处理过的反馈会转为用户通知

#### 用户蓝图 `app/user/user.py`

负责：

- 更新邮箱与备注
- 充值
- 修改密码
- 检查是否有未完成订单
- 注销账号

#### 管理员蓝图 `app/admin/device.py`

负责设备增删改查：

- `GET /api/devices`
- `POST /api/devices`
- `PUT /api/devices/<id>`
- `DELETE /api/devices/<id>`

#### 管理员蓝图 `app/admin/pricing.py`

负责计费规则：

- `GET /api/pricing-rules`
- `POST /api/pricing-rules`

特点：

- 保存后不仅更新数据库
- 还会调用链上 `PowerManager` 更新计费信息

#### 管理员蓝图 `app/admin/dispute.py`

负责：

- 获取全部反馈
- 获取工单关联订单详情

#### 管理员蓝图 `app/admin/visual.py`

负责看板统计：

- 总营收
- 热门地点
- 总订单量
- 平均使用时长
- 营收趋势图
- 地点分布饼图

### 5.6 数据库层 `app/extensions.py`

后端没有使用 SQLAlchemy，而是封装了一个基于 `DBUtils.PooledDB` 的连接池：

- 统一暴露 `execute / fetchone / fetchall`
- 支持启动时自动执行 SQL 文件
- 业务层直接写 SQL

优点是直接、清晰；缺点是字段变更时需要特别注意前后端与 SQL 的一致性。

## 7. 区块链与系统管理层：`app/sysadmin`

这是本项目区别于普通 CRUD 系统的关键部分。

### 7.1 `conf.py`

定义了：

- `FISCO_SDK_PATH`
- MySQL 配置
- 多个合约地址与 ABI 路径

可见的合约包括：

- `AdminManager`
- `FeedbackManager`
- `PowerManager`
- `RentRecordManager`
- `UserStorage`

### 7.2 `bank.py`

功能：

- 用 Selenium 打开 WeBASE 页面
- 无头模式抓取 `sysadmin` 对应地址

### 7.3 `synchronization.py`

功能：

- 从 WeBASE 拉取合约信息
- 更新本地合约 ABI / bin / address
- 必要时重新部署合约
- 把新的合约地址再注册回 WeBASE

可以理解为“合约元数据同步器”。

### 7.4 `sync_worker.py`

功能：

- 持续执行链下数据库与链上合约的数据同步
- 双向同步表：
  - `admin`
  - `users`
  - `billing_rules`
  - `Information`
- 单向上链表：
  - `feedback`
  - `rent_record`

这是整个系统里最复杂的一层，也是真正体现“业务系统 + 区块链同步”特点的部分。

## 8. 核心业务流程

### 8.1 注册流程

1. 前端提交用户名、邮箱、密码
2. 后端写入本地 `users`
3. 后端调用链上用户服务创建链上用户
4. 注册完成

### 8.2 登录流程

#### 普通用户

1. 前端传 `role=users`
2. 后端按用户地址查询 `users`
3. 返回昵称、邮箱、余额、备注等信息
4. 前端写入 `localStorage`

#### 管理员

1. 前端传 `role=admin`
2. 后端查询 `admin.address`
3. 返回权限对象
4. 前端按权限决定可进入的后台模块

#### 系统管理员

1. 前端传 `role=sysadmin`
2. 后端读取 WeBASE 中的系统管理员地址
3. 命中后返回跳转信息
4. 前端跳往 WeBASE 页面

### 8.3 租借流程

1. 用户在 `RentService.vue` 获取设备列表
2. 点击租借后调用 `POST /api/rent`
3. 后端检查用户是否已有未归还设备
4. 后端检查余额是否足够支付押金
5. 扣除押金
6. 更新 `Information`
7. 写入 `rent_record`

### 8.4 归还 / 结算流程

系统实际上存在两条归还思路：

- 在设备页调用 `POST /api/return`
- 在订单页调用 `POST /api/history` 完成结算

两者都会：

- 读取订单开始时间
- 根据计费规则计算费用
- 释放设备占用
- 更新订单状态
- 退还押金并扣除实际费用

### 8.5 反馈处理流程

1. 用户在订单页提交反馈
2. 反馈写入 `feedback`
3. 管理员后台读取工单
4. 管理员提交回复并把状态设为 `processed`
5. 用户通过 `/api/notifications` 看到新通知

## 10. 当前实现特点与风险

### 10.1 优点

- 前后端职责分层比较清晰
- 用户端、管理端、系统管理员端分层明确
- 数据库表结构与业务页面基本一一对应
- 链上同步模块有实际落地代码，不是空壳

### 10.2 当前实现特点

- 前端以 `localStorage` 为主，没有完整 token 认证闭环
- 后端大量使用原生 SQL
- 区块链同步逻辑较重，后端启动成本高于普通 Flask 项目
- 项目同时包含业务系统和链上同步两套关注点

### 10.3 需要重点注意的工程风险

- 前后端对“用户名 / 地址 / 昵称”的字段语义存在一定混用
- 登录接口中普通用户和管理员当前密码校验较弱
- 前端请求拦截器预留了 token，但后端没有完整 token 流程
- 归还与结算存在两套接口路径，后续更适合统一
- 区块链相关地址和 WeBASE 地址对环境比较敏感，迁移时需要重点检查
